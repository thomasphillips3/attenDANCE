# Plan 05-04: Event/Recital Management + Costume Tracking + Phase Verification

## Goal
Admin can create a recital or event, enroll students, and track costume status per student (not ordered through returned). Final phase verification confirms all Phase 5 success criteria are met.

## Requirements
- EVNT-01: Event/recital CRUD
- EVNT-02: Event enrollment
- EVNT-03: Costume tracking

## Dependencies
- Plan 05-01 (admin dashboard and sidebar navigation)
- Existing schema: events, event_enrollments, costumes tables already exist from migration 000100

## Tasks

### Task 1: Events API endpoints
**File:** `server/src/routes/events.ts`

Create event management endpoints (admin-only):

1. `GET /events` — List events with query params: `upcoming` (boolean), `type` filter. Include enrolled student count per event.
2. `GET /events/:id` — Event detail with enrolled students and their costume status.
3. `POST /events` — Create event: `{ name, event_date, venue, type }`.
4. `PUT /events/:id` — Update event.
5. `DELETE /events/:id` — Delete event (cascade deletes enrollments and costumes).

6. `POST /events/:id/enroll` — Enroll student(s): `{ student_ids: uuid[] }`. Bulk enroll.
7. `DELETE /events/:id/enroll/:studentId` — Remove student from event.

8. `GET /events/:id/costumes` — List costumes for event with student names.
9. `POST /events/:id/costumes` — Create costume entry: `{ student_id, description, size }`.
10. `PUT /costumes/:id` — Update costume status (ordered, received, paid booleans).

Register plugin in `server/src/index.ts`.

### Task 2: Events RLS policies
**File:** `supabase/migrations/20260522001100_events_rls.sql`

Add RLS policies for events, event_enrollments, and costumes tables:
- Admin: full CRUD on all three tables, scoped to organization_id
- Instructor: read-only on events and event_enrollments
- Parent: read-only on event_enrollments for their family's students

Use the subselect pattern for auth.jwt() organization_id.
