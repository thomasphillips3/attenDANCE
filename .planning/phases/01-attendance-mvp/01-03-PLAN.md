PATCH /attendance endpoint contract:
  URL: /attendance
  Method: PATCH
  Auth: required (Bearer JWT via existing preHandler from Plan 01)
  Required header: X-Idempotency-Key: {clientId — a UUID string}
  Request body: { sessionId: string (uuid), studentId: string (uuid), status: 'present'|'absent'|'late'|'excused' }
  Server behavior (in order):
    1. Extract organizationId from request.organizationId (JWT — never from body)
    2. Read X-Idempotency-Key header — return 400 if missing
    3. Check in-memory idempotency Map: if clientId present and entry age < 1 hour, return 200 { message: 'already processed', attendanceId: null, status: body.status }
    4. Validate body with TypeBox schema — return 400 on validation error
    5. Look up staffId: SELECT id FROM staff WHERE user_id = request.user.id AND organization_id = $orgId
    6. Upsert via fastify.supabase (service role):
       INSERT INTO attendance_records (id, organization_id, student_id, class_session_id, status, marked_by, marked_by_staff_id)
       VALUES (gen_random_uuid(), $org, $studentId, $sessionId, $status, 'manual', $staffId)
       ON CONFLICT (student_id, class_session_id) DO UPDATE
         SET status = EXCLUDED.status, marked_by = 'manual', updated_at = now()
       RETURNING id
    7. Store in idempotency Map: idempotencyStore.set(clientId, Date.now())
    8. Return 200 { attendanceId: returned_id, status: body.status }
    9. On DB error: return 500 { error: 'Failed to record attendance' }

  Idempotency store: module-level const idempotencyStore = new Map<string, number>()
    Value is Date.now() at processing time. Entries older than 3_600_000ms are ignored (1 hour TTL).
    The ON CONFLICT (student_id, class_session_id) DO UPDATE is the authoritative dedup.
    The Map is the fast-path to skip the DB round-trip on rapid retries.

From RESEARCH.md Pattern 7 — optimistic mark sequence in client/src/screens/Roster.tsx:
  onMark = async (studentId: string, status: AttendanceStatus) => {
    // Step 1: immediate local UI update — no await
    setLocalStatus(prev => ({ ...prev, [studentId]: status }))
    // Step 2: generate idempotency key stamped NOW
    const clientId = crypto.randomUUID()
    // Step 3: durable IndexedDB write — must happen before any fetch
    await db.attendance_queue.add({
      clientId, studentId, sessionId, status,
      createdAt: Date.now(),  // CRITICAL: stamp at tap time, not sync time
      synced: false, retries: 0
    })
    // Step 4: fire-and-forget API call if online
    if (navigator.onLine) {
      ;(async () => {
        try {
          const res = await fetch(`${VITE_API_URL}/attendance`, {
            method: 'PATCH',
            headers: {
              'Content-Type': 'application/json',
              'Authorization': `Bearer ${token}`,
              'X-Idempotency-Key': clientId,
            },
            body: JSON.stringify({ sessionId, studentId, status }),
          })
          if (res.ok || res.status === 409) {
            await db.attendance_queue.where('clientId').equals(clientId).modify({ synced: true })
          }
        } catch { /* stays in queue, drainQueue retries on reconnect */ }
      })()
    }
  }

From client/src/lib/sync.ts (Plan 02 stub — replace entirely with full implementation):
  drainQueue(token: string): Promise<void>
    Reads: db.attendance_queue.where('synced').equals(0).sortBy('createdAt')  -- chronological ASC
    For each item:
      fetch PATCH {VITE_API_URL}/attendance with headers:
        Authorization: Bearer {token}
        X-Idempotency-Key: {item.clientId}
        Content-Type: application/json
      body: { sessionId: item.sessionId, studentId: item.studentId, status: item.status }
      On 200 or 409: db.attendance_queue.update(item.id!, { synced: true })
      On 4xx (not 409): db.attendance_queue.update(item.id!, { retries: 99 }) — log with console.warn, do not retry
      On 5xx or network error: db.attendance_queue.update(item.id!, { retries: item.retries + 1 })
    After full drain: call queryClient.invalidateQueries({ queryKey: ['roster'] })

  useSyncOnReconnect(token: string | undefined): { pendingCount: number }
    pendingCount: useLiveQuery(() => db.attendance_queue.where('synced').equals(0).count(), [], 0)
    useEffect: window.addEventListener('online', handler) + document.addEventListener('visibilitychange', handler)
      handler: if (document.visibilityState === 'visible' && navigator.onLine && token) drainQueue(token)
    On mount: if (navigator.onLine && token) drainQueue(token) immediately
    Cleanup: removeEventListener on unmount
    No SyncManager. No BackgroundSync. No sync.register. iOS foreground sync only.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser PATCH /attendance -> Fastify | JWT required; organizationId from JWT only; X-Idempotency-Key from request header |
| IndexedDB queue -> PATCH /attendance | clientId UUID used as idempotency key; server has both in-memory Map fast-path and DB-level ON CONFLICT dedup |
| Offline queue ordering -> server | createdAt stamped at mark time guarantees chronological replay order |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Duplicate attendance writes on retry | mitigate | In-memory Map keyed by clientId provides fast-path dedup. DB-level ON CONFLICT (student_id, class_session_id) DO UPDATE is the authoritative guard — at most one row per student per session. |
| T-03-02 | Tampering | Offline queue ordering corrupts attendance state | mitigate | createdAt stamped inside onMark at tap time. drainQueue sorts by createdAt ASC. Preserves edit sequence (e.g. absent then corrected to present arrives in correct order). Per PITFALLS.md Pitfall 1. |
| T-03-03 | Spoofing | Client supplies organizationId in PATCH body | mitigate | PATCH /attendance ignores any organization_id in the request body. organizationId taken exclusively from request.organizationId set by the auth preHandler from Plan 01. |
| T-03-04 | Information Disclosure | marked_by_staff_id exposure | accept | Staff IDs are opaque UUIDs with no PII. RLS-gated — other tenants cannot read. Stored for audit trail only. |
| T-03-SC | Tampering | npm installs | mitigate | No new npm packages in this plan. All packages audited in 01-RESEARCH.md. |
</threat_model>
